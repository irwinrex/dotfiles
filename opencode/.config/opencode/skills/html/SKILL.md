## Use when

- Working on HTML, CSS, templates, static pages, forms, layout, accessibility, SEO metadata, or rendered UI structure.
- Reviewing markup generated by frameworks or server-side templates.

## Rules

- Prefer semantic HTML before JavaScript-driven behavior.
- Keep accessibility explicit: labels, landmarks, headings, focus states, keyboard paths, alt text, contrast, and ARIA only when needed.
- Do not break progressive enhancement. Core content and forms should remain understandable without client-side JavaScript when practical.
- Avoid inline styles unless matching existing patterns or required for email/templates.
- Preserve existing design tokens, CSS architecture, and component conventions.

## Verification

- Check rendered structure mentally or with existing preview/build commands.
- For forms, verify labels, validation messages, autocomplete, disabled/loading states, and error recovery.
- For layout, check responsive behavior and overflow risk.

## DevOps checks

- Confirm static asset paths, cache-busting, CSP-sensitive inline scripts/styles, and environment-specific base URLs.
- Do not embed secrets, internal URLs, or environment-specific credentials in markup.
